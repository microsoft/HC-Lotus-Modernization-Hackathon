2Purpose & Scope

2.1 Purpose

This Standard Operating Procedure (SOP) establishes the governance framework, acceptable use guidelines, data handling requirements, and oversight mechanisms for the deployment and use of GitHub Copilot as an AI-assisted coding tool within the organization.

GitHub Copilot is an AI-powered developer assistant — powered by large language models (LLMs) — that provides in-editor code suggestions, completions, and explanations. This SOP ensures that its use aligns with Government of Canada (GC) policy obligations, information security requirements, and the principles of responsible AI.

2.2 Scope

This SOP applies to:

  • All employees, contractors, and consultants authorized to use GitHub Copilot under the organizational enterprise license
  • All source code repositories, integrated development environments (IDEs), and development pipelines where GitHub Copilot is activated
  • Use of GitHub Copilot in GitHub.com cloud environments, GitHub Enterprise Server, and any IDE plugins (VS Code, Visual Studio, JetBrains, Neovim)
⚠️ Out of Scope This SOP does not govern use of consumer-grade AI coding tools (e.g., ChatGPT, Claude, Gemini) for development purposes. Use of such tools is addressed in the organization's separate Generative AI Acceptable Use Policy.

2.3 Objectives

  • Enable developer productivity through AI assistance while managing risk
  • Ensure compliance with GC information security and privacy obligations
  • Establish clear roles, responsibilities, and accountability
  • Protect against intellectual property, supply chain, and data residency risks
  • Support responsible AI principles including human oversight and transparency

5Acceptable Use Policy

5.1 Authorized Uses

GitHub Copilot may be used for the following activities on approved government workloads:

  • Generating code completions and suggestions in authorized repositories
  • Explaining existing code, generating inline documentation and comments
  • Writing unit tests, test fixtures, and boilerplate configuration files
  • Refactoring and modernizing legacy code (e.g., Powerbuilder modernization PoC)
  • Generating SQL queries, scripts, and pipeline definitions on non-production data
  • Answering developer questions in Copilot Chat about APIs, syntax, and architecture
  • Generating draft commit messages and pull request descriptions

5.2 Prohibited Uses

🚫 PROHIBITED — Non-Compliant Uses The following uses are expressly prohibited and may result in disciplinary action and/or license revocation:
  • Inputting Protected B or higher data into any Copilot prompt or context window — including voter registration data, financial data, health information, law enforcement data, or security credentials
  • Inputting Personally Identifiable Information (PII) of Canadian citizens or employees
  • Using Copilot on repositories containing classified information (Protected B, C, or Secret)
  • Accepting AI-generated code without review, testing, and validation
  • Using Copilot to circumvent security controls, generate exploit code, or probe vulnerabilities
  • Sharing Copilot-generated code externally without review for intellectual property concerns
  • Using personal GitHub accounts to access Copilot on government workloads
  • Installing Copilot plugins on non-approved, non-managed devices

5.3 Developer Obligations

Every authorized developer agrees to the following upon license activation:

  • Complete mandatory training before first use (see Section 7.2)
  • Treat all Copilot suggestions as untrusted draft code requiring human review
  • Flag AI-assisted contributions in pull request descriptions using the standard tag: [copilot-assisted]
  • Immediately report suspected data leakage, inappropriate suggestions, or security concerns to IT Security
  • Not circumvent organization-level policy settings (e.g., re-enabling disabled features)

6Data Classification & Handling Requirements

GitHub Copilot processes code context (open files, surrounding code) as prompt input to generate suggestions. The following matrix governs what data types may be included in this context:

Classification Examples (Government Context) GitHub Copilot Controls Required
UNCLASSIFIED Public-facing code, open-source libraries, generic business logic, configuration templates with no sensitive values ✅ Permitted Standard enterprise controls; training data opt-out enabled (see 9.2)
PROTECTED A Internal business processes, low-sensitivity employee information, general correspondence templates ⚠️ Conditional Data must be generalized/anonymized before use as prompt context; SA&A must confirm acceptable residency; no hardcoded credentials
PROTECTED B Electoral data, voter registration records, financial processing code with live data, employee SINs, health records, law enforcement data, political finance records 🚫 PROHIBITED Copilot must be disabled in any IDE workspace or repository containing Protected B data. Contact IT Security for approved alternatives.

6.1 Repository-Level Controls

GitHub Enterprise Copilot policies shall be configured at the organization level to:

  • Enable Copilot only for explicitly approved repositories (allowlist model)
  • Disable Copilot in repositories tagged or classified as containing Protected A or B data
  • Require repository-level .copilotignore files in all repositories containing any sensitive paths, configuration files, or credential stores
💡 .copilotignore Best Practice A .copilotignore file (similar to .gitignore) tells GitHub Copilot which files and directories to exclude from its context window. Recommended exclusions include: .env, *.pem, *.key, secrets/, config/credentials/, any files containing database connection strings or API keys, and any files containing test data derived from production systems.

6.2 Personal Information — Privacy Act Obligations

Under the Privacy Act (R.S.C., 1985, c. P-21), personal information about identifiable individuals must not be disclosed to third-party systems without consent and legal authority. GitHub Copilot — even with training data opt-out enabled — should not process source code files or query contexts that contain:

  • Names, SINs, dates of birth, addresses, or other personal identifiers
  • Voter registration or electoral roll data
  • Any data collected under the Canada Elections Act or equivalent provincial statutes
  • Employee personnel records

Developers must ensure all test data used in Copilot-accessible workspaces is synthetic or properly anonymized.

7Licensing, Onboarding & Access Management

7.1 Prerequisites — Security Assessment & Authorization (SA&A)

In accordance with the GC Directive on Security Management and ITSG-33, a Security Assessment and Authorization (SA&A) must be completed and accepted prior to enterprise-wide rollout of GitHub Copilot. The SA&A shall assess:

  • Data residency and sovereignty (GitHub Copilot uses Microsoft Azure data centres; confirm region alignment with GC requirements)
  • Transmission security (TLS 1.3 in transit; no persistent storage of prompts under enterprise agreement)
  • Training data opt-out verification
  • Identity and access management integration (SSO / Azure AD / Entra ID)
  • Audit logging and SIEM integration capability
  • Supply chain security (GitHub Advanced Security dependency review)
✅ GitHub Copilot Enterprise — Key Security Commitments Under the GitHub Copilot Enterprise Agreement: (1) Prompts and suggestions are not used to train GitHub's models; (2) Code context is not stored beyond the session; (3) Telemetry is configurable and can be disabled; (4) Data processed in Azure datacentres (US East / Canada available); (5) SOC 2 Type II certified; ISO 27001 compliant.

7.2 License Provisioning Process

  1. Developer's manager submits license request through ACTION REQUIRED — insert ITSM/ServiceNow workflow name
  2. IT Security verifies the developer's workload does not involve Protected B data
  3. GitHub Copilot Administrator assigns seat via GitHub Enterprise organization console
  4. Developer receives activation email and link to mandatory training

8Code Review & Human Oversight Requirements

🍁 Directive on Automated Decision-Making — Human in the Loop GitHub Copilot suggestions constitute automated decision support. Consistent with the GC Directive on Automated Decision-Making, no AI-generated code shall be merged to a production branch without explicit human review and approval by an authorized developer or technical lead.

8.1 Pull Request Requirements

All pull requests containing AI-assisted code must:

  • Include the tag [copilot-assisted] in the PR title or description
  • Receive at least one approved review from a developer who was not the author (i.e., not the same individual who accepted the Copilot suggestion)
  • Pass all automated CI/CD pipeline checks including static analysis, unit tests, and dependency vulnerability scans (GitHub Advanced Security)
  • For Protected A repositories: require a second reviewer sign-off from the team lead or security reviewer

8.2 Developer Responsibility

Developers remain fully accountable for any code they commit, regardless of whether it was AI-generated. Accepting a Copilot suggestion is equivalent to authoring the code. Developers must:

  • Understand every line of AI-generated code before committing
  • Validate logic correctness, edge cases, and error handling
  • Ensure no sensitive data (credentials, PII) was inadvertently included in a suggestion
  • Run and pass all relevant unit and integration tests
  • Review Copilot suggestions for potential copyright issues (see Section 9.3)

8.3 Prohibited Automation Patterns

  • Auto-merge of AI-generated code without human review is expressly prohibited
  • Using Copilot to generate scripts that automate access to government systems without proper authorization
  • Using Copilot Workspace to auto-generate and push entire feature branches without developer review of each commit

9Security, Supply Chain & Intellectual Property

9.1 Secrets & Credential Detection

GitHub Advanced Security (GHAS) must be enabled on all repositories where Copilot is active. GHAS Secret Scanning provides real-time detection of accidentally committed credentials. Organizations shall:

  • Enable GitHub Secret Scanning with push protection enabled (blocks pushes containing detected secrets)
  • Enable Dependabot alerts and security updates for all dependencies
  • Enforce code scanning (CodeQL or equivalent) as a required status check before merge
  • Conduct quarterly reviews of all GHAS alerts and resolution timelines

9.2 Training Data Opt-Out (Enterprise Configuration)

Under the GitHub Copilot Enterprise agreement, code prompts and suggestions are not used to train GitHub's models by default. The GitHub Copilot Administrator must confirm this setting in the organization's Copilot policy console:

  • Navigate to: GitHub Enterprise → Organization → Settings → Copilot → Policies
  • Confirm: "Allow GitHub to use my code snippets from the code editor to improve GitHub Copilot" is set to Disabled
  • Screenshot this configuration and retain as evidence for SA&A and audit purposes

9.3 Intellectual Property & Copyright

GitHub Copilot may occasionally suggest code that is similar to existing open-source code. Developers must be aware:

  • Enable GitHub Copilot's built-in duplication detection filter — this flags suggestions that closely match public code and displays the matching licence
  • Do not accept suggestions flagged as matching GPL, AGPL, or other copyleft licences for use in proprietary government systems without legal review
  • All AI-generated code incorporated into Crown Copyright works remains subject to the Copyright Act — the Crown retains copyright; developers should document significant AI-assisted contributions
  • Do not submit government code to Copilot prompts in a way that discloses unreleased or proprietary business logic

9.4 Supply Chain Security

Copilot may suggest third-party library imports. Before accepting such suggestions:

  • Verify the library against the organization's approved dependency catalogue
  • Check for known CVEs via Dependabot or NVD before merging
  • Do not add new package dependencies without team lead approval

10Official Languages Requirements

🇨🇦 Official Languages Act Obligations Federal institutions subject to the Official Languages Act (R.S.C., 1985, c. 31) must ensure that AI-generated content used in public-facing or employee-facing communications is available in both English and French.

10.1 Code Comments & Documentation

GitHub Copilot generates code comments and documentation primarily in English (the default). For documentation, inline comments, and user-facing strings used in production systems, teams must:

  • Review AI-generated comments for accuracy before committing — do not rely on Copilot for bilingual content generation without human review
  • For any user-facing strings or system messages generated by Copilot, ensure French translations are produced and reviewed by qualified translators or bilingual staff
  • Use established translation workflows; Copilot-generated French translations are not considered an official translation for ATIP or public communications purposes

10.2 Internal Technical Documentation

Internal technical documentation (README files, architecture notes, developer wikis) generated with Copilot assistance may be maintained in English for internal developer use, consistent with the institution's internal language of work policy.

11Monitoring, Reporting & Audit

11.1 Usage Metrics (GitHub Copilot Dashboard)

The GitHub Copilot Administrator shall produce a quarterly usage report to the CIO/CISO, including:

  • Total active seats vs. licensed seats (utilization rate)
  • Copilot suggestion acceptance rate by team/repository
  • Copilot Chat query volumes
  • Any incidents or flagged misuse events
  • Training completion rates and outstanding recertifications

11.2 Audit Log Retention

GitHub Enterprise audit logs capture all administrative actions (seat assignments, policy changes, feature enablement). These logs shall be:

  • Retained for a minimum of two (2) years consistent with GC information management obligations
  • Exported monthly to the organization's SIEM or secure log archive
  • Available for ATIP disclosure requests, security reviews, and internal audits

11.3 Annual Security Review

IT Security shall conduct an annual review of the GitHub Copilot deployment, including:

  • Reassessment of data residency and service terms (vendor agreements change)
  • Review of GHAS alerts and unresolved vulnerabilities introduced since last review
  • Review of any new Copilot features enabled during the year and their security implications
  • Updated threat and risk assessment if the organizational risk posture has changed

12Incident Response

12.1 Reportable Incidents

The following events must be reported immediately to IT Security via ACTION REQUIRED — insert reporting channel/ticket queue:

  • Suspected submission of Protected B or PII data to Copilot (accidental prompt injection)
  • Copilot suggesting what appears to be sensitive internal data from another repository or system
  • Malicious code suggestions (e.g., suggestions containing obvious backdoors, credential exfiltration)
  • Unauthorized use of Copilot by an unprovisioned individual
  • Loss or compromise of a developer device with an active Copilot session

12.2 Response Procedures

  1. Contain: Immediately revoke the affected developer's Copilot license and GitHub access if a breach is suspected
  2. Assess: IT Security reviews Copilot audit logs and GitHub activity to determine scope
  3. Notify: If personal information may have been disclosed, Privacy Officer notified within 24 hours; OPC notification as required under the Privacy Act
  4. Remediate: Remove any committed code containing sensitive data; rotate exposed credentials; issue GitHub secret scanning alerts
  5. Document: Incident report filed within 72 hours; lessons learned reviewed within 30 days
