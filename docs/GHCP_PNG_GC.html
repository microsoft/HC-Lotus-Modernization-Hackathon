2Objet et portée

2.1 Objet

Cette Procédure normalisée d'exploitation (PNE) établit le cadre de gouvernance, les lignes directrices d'utilisation acceptable, les exigences de traitement des données et les mécanismes de supervision pour le déploiement et l'utilisation de GitHub Copilot en tant qu'outil de codage assisté par l'IA au sein de l'organisation.

GitHub Copilot est un assistant de développement alimenté par l'IA — propulsé par des grands modèles de langage (GML) — qui fournit des suggestions, complétions et explications de code dans l'éditeur. Cette PNE garantit que son utilisation est conforme aux obligations politiques du gouvernement du Canada (GC), aux exigences en matière de sécurité de l'information et aux principes d'IA responsable.

2.2 Portée

Cette PNE s'applique à :

  • Tous les employés, entrepreneurs et consultants autorisés à utiliser GitHub Copilot dans le cadre de la licence d'entreprise organisationnelle
  • Tous les référentiels de code source, environnements de développement intégrés (EDI) et pipelines de développement où GitHub Copilot est activé
  • L'utilisation de GitHub Copilot dans les environnements cloud GitHub.com, GitHub Enterprise Server et tout plugin IDE (VS Code, Visual Studio, JetBrains, Neovim)
⚠️ Hors de portée Cette PNE ne régit pas l'utilisation d'outils de codage IA grand public (p. ex., ChatGPT, Claude, Gemini) à des fins de développement. L'utilisation de ces outils est traitée dans la Politique d'utilisation acceptable de l'IA générative distincte de l'organisation.

2.3 Objectifs

  • Favoriser la productivité des développeurs grâce à l'assistance IA tout en gérant les risques
  • Assurer la conformité avec les obligations du GC en matière de sécurité de l'information et de protection de la vie privée
  • Établir des rôles, des responsabilités et une reddition de comptes clairs
  • Protéger contre les risques liés à la propriété intellectuelle, à la chaîne d'approvisionnement et à la résidence des données
  • Soutenir les principes d'IA responsable, notamment la supervision humaine et la transparence

4.1 Administrateur GitHub Copilot

Un rôle désigné d'Administrateur GitHub Copilot doit être attribué au niveau de l'entreprise. Cette personne est responsable de la gestion des attributions de sièges, de la configuration des politiques organisationnelles, de l'activation et de la désactivation des fonctionnalités Copilot, et de la production de rapports d'utilisation. Ce rôle devrait relever de la fonction de sécurité des TI ou de transformation numérique. ACTION REQUISE — attribuer un responsable nommé

5Politique d'utilisation acceptable

5.1 Utilisations autorisées

GitHub Copilot peut être utilisé pour les activités suivantes sur les charges de travail gouvernementales approuvées :

  • Générer des complétions et suggestions de code dans les référentiels autorisés
  • Expliquer le code existant, générer de la documentation en ligne et des commentaires
  • Rédiger des tests unitaires, des appareils de test et des fichiers de configuration passe-partout
  • Refactoriser et moderniser le code patrimonial (p. ex., modernisation Powerbuilder PoC)
  • Générer des requêtes SQL, des scripts et des définitions de pipeline sur des données hors production
  • Répondre aux questions des développeurs dans Copilot Chat sur les API, la syntaxe et l'architecture
  • Générer des messages de commit et des descriptions de demandes de tirage

5.2 Utilisations interdites

🚫 INTERDIT — Utilisations non conformes Les utilisations suivantes sont expressément interdites et peuvent entraîner des mesures disciplinaires et/ou la révocation de la licence :
  • Saisir des données Protégé B ou de niveau supérieur dans toute invite ou fenêtre de contexte Copilot — y compris les données d'inscription des électeurs, les données financières, les renseignements sur la santé, les données des forces de l'ordre ou les identifiants de sécurité
  • Saisir des renseignements personnels identifiables (RPI) de citoyens canadiens ou d'employés
  • Utiliser Copilot sur des référentiels contenant des informations classifiées (Protégé B, C ou Secret)
  • Accepter du code généré par l'IA sans révision, test et validation
  • Utiliser Copilot pour contourner les contrôles de sécurité, générer du code d'exploitation ou sonder les vulnérabilités
  • Partager du code généré par Copilot en externe sans révision pour les problèmes de propriété intellectuelle
  • Utiliser des comptes GitHub personnels pour accéder à Copilot sur des charges de travail gouvernementales
  • Installer des plugins Copilot sur des appareils non approuvés et non gérés

5.3 Obligations des développeurs

Chaque développeur autorisé accepte ce qui suit à l'activation de la licence :

  • Compléter la formation obligatoire avant la première utilisation (voir Section 7.2)
  • Traiter toutes les suggestions de Copilot comme du code brouillon non fiable nécessitant une révision humaine
  • Signaler les contributions assistées par l'IA dans les descriptions de demandes de tirage à l'aide de la balise standard : [copilot-assisted]
  • Signaler immédiatement toute fuite de données suspecte, suggestion inappropriée ou problème de sécurité à la Sécurité des TI
  • Ne pas contourner les paramètres de politique au niveau de l'organisation (p. ex., réactiver des fonctionnalités désactivées)

6Classification des données et exigences de traitement

GitHub Copilot traite le contexte du code (fichiers ouverts, code environnant) comme entrée d'invite pour générer des suggestions. La matrice suivante régit les types de données pouvant être inclus dans ce contexte :

Classification Exemples (contexte gouvernemental) GitHub Copilot Contrôles requis
NON CLASSIFIÉ Code destiné au public, bibliothèques open source, logique métier générique, modèles de configuration sans valeurs sensibles ✅ Autorisé Contrôles d'entreprise standard; désinscription aux données d'entraînement activée (voir 9.2)
PROTÉGÉ A Processus commerciaux internes, renseignements d'employés à faible sensibilité, modèles de correspondance générale ⚠️ Conditionnel Les données doivent être généralisées/anonymisées avant utilisation comme contexte d'invite; l'ES&A doit confirmer la résidence acceptable; pas d'identifiants codés en dur
PROTÉGÉ B Données électorales, registres d'inscription des électeurs, code de traitement financier avec données en direct, NAS des employés, dossiers de santé, données des forces de l'ordre, dossiers financiers politiques 🚫 INTERDIT Copilot doit être désactivé dans tout espace de travail IDE ou référentiel contenant des données Protégé B. Contactez la Sécurité des TI pour les alternatives approuvées.

6.1 Contrôles au niveau du référentiel

Les politiques GitHub Enterprise Copilot doivent être configurées au niveau de l'organisation pour :

  • Activer Copilot uniquement pour les référentiels explicitement approuvés (modèle de liste d'autorisation)
  • Désactiver Copilot dans les référentiels étiquetés ou classifiés comme contenant des données Protégé A ou B
  • Exiger des fichiers .copilotignore au niveau du référentiel dans tous les référentiels contenant des chemins sensibles, des fichiers de configuration ou des magasins d'identifiants
💡 Bonne pratique .copilotignore Un fichier .copilotignore (similaire à .gitignore) indique à GitHub Copilot quels fichiers et répertoires exclure de sa fenêtre de contexte. Les exclusions recommandées incluent : .env, *.pem, *.key, secrets/, config/credentials/, tout fichier contenant des chaînes de connexion de base de données ou des clés API, et tout fichier contenant des données de test dérivées des systèmes de production.

6.2 Renseignements personnels — Obligations en vertu de la Loi sur la protection des renseignements personnels

En vertu de la Loi sur la protection des renseignements personnels (L.R.C., 1985, ch. P-21), les renseignements personnels sur les personnes identifiables ne doivent pas être divulgués à des systèmes tiers sans consentement et autorité légale. GitHub Copilot — même avec la désinscription aux données d'entraînement activée — ne doit pas traiter les fichiers de code source ou les contextes de requête contenant :

  • Noms, NAS, dates de naissance, adresses ou autres identifiants personnels
  • Données d'inscription des électeurs ou de liste électorale
  • Toute donnée recueillie en vertu de la Loi électorale du Canada ou des lois provinciales équivalentes
  • Dossiers du personnel des employés

Les développeurs doivent s'assurer que toutes les données de test utilisées dans les espaces de travail accessibles par Copilot sont synthétiques ou correctement anonymisées.

7Licences, intégration et gestion des accès

7.1 Prérequis — Évaluation et autorisation de sécurité (ES&A)

Conformément à la Directive sur la gestion de la sécurité du GC et à l'ITSG-33, une évaluation et autorisation de sécurité (ES&A) doit être complétée et acceptée avant le déploiement à l'échelle de l'entreprise de GitHub Copilot. L'ES&A doit évaluer :

  • La résidence et la souveraineté des données (GitHub Copilot utilise les centres de données Microsoft Azure; confirmer l'alignement régional avec les exigences du GC)
  • La sécurité de transmission (TLS 1.3 en transit; pas de stockage persistant des invites dans le cadre de l'accord d'entreprise)
  • La vérification de la désinscription aux données d'entraînement
  • L'intégration de la gestion des identités et des accès (SSO / Azure AD / Entra ID)
  • La journalisation des audits et la capacité d'intégration SIEM
  • La sécurité de la chaîne d'approvisionnement (révision des dépendances GitHub Advanced Security)
✅ GitHub Copilot Enterprise — Engagements clés en matière de sécurité Dans le cadre de l'accord GitHub Copilot Enterprise : (1) Les invites et suggestions ne sont pas utilisées pour entraîner les modèles de GitHub; (2) Le contexte du code n'est pas stocké au-delà de la session; (3) La télémétrie est configurable et peut être désactivée; (4) Données traitées dans les centres de données Azure (Est des États-Unis / Canada disponible); (5) Certifié SOC 2 Type II; conforme ISO 27001.

7.2 Processus d'attribution de licences

  1. Le gestionnaire du développeur soumet une demande de licence via ACTION REQUISE — insérer le nom du flux de travail ITSM/ServiceNow
  2. La Sécurité des TI vérifie que la charge de travail du développeur n'implique pas de données Protégé B
  3. L'administrateur GitHub Copilot attribue un siège via la console de l'organisation GitHub Enterprise
  4. Le développeur reçoit un courriel d'activation et un lien vers la formation obligatoire
  5. Le développeur complète la formation et prend connaissance de cette PNE (suivi dans ACTION REQUISE — insérer le système de gestion de l'apprentissage)
  6. Le développeur active Copilot dans les IDE approuvés en utilisant les identifiants SSO organisationnels

7.4 Révocation de l'accès

Les licences GitHub Copilot seront révoquées dans les conditions suivantes :

  • Départ ou changement de rôle de l'employé ou entrepreneur supprimant les responsabilités de développement
  • Violation confirmée de la PNE (à la discrétion du gestionnaire et de la Sécurité des TI)
  • Défaut de compléter la recertification annuelle dans le délai désigné
  • Compromission de l'appareil ou accès non autorisé suspecté

8Révision du code et exigences de supervision humaine

🍁 Directive sur la prise de décision automatisée — Intervention humaine Les suggestions de GitHub Copilot constituent une aide à la décision automatisée. Conformément à la Directive sur la prise de décision automatisée du GC, aucun code généré par IA ne doit être fusionné vers une branche de production sans révision et approbation humaine explicites par un développeur autorisé ou un chef technique.

8.1 Exigences relatives aux demandes de tirage

Toutes les demandes de tirage contenant du code assisté par IA doivent :

  • Inclure la balise [copilot-assisted] dans le titre ou la description de la demande de tirage
  • Recevoir au moins une révision approuvée d'un développeur qui n'est pas l'auteur (c.-à-d. pas le même individu qui a accepté la suggestion Copilot)
  • Passer toutes les vérifications du pipeline CI/CD automatisé, y compris l'analyse statique, les tests unitaires et les analyses de vulnérabilité des dépendances (GitHub Advanced Security)
  • Pour les référentiels Protégé A : exiger la signature d'un second réviseur du chef d'équipe ou du réviseur de sécurité

8.2 Responsabilité des développeurs

Les développeurs restent entièrement responsables de tout code qu'ils soumettent, qu'il soit généré par IA ou non. Accepter une suggestion Copilot est équivalent à rédiger le code. Les développeurs doivent :

  • Comprendre chaque ligne de code généré par IA avant de s'engager
  • Valider la correction logique, les cas limites et la gestion des erreurs
  • S'assurer qu'aucune donnée sensible (identifiants, RPI) n'a été inadvertamment incluse dans une suggestion
  • Exécuter et réussir tous les tests unitaires et d'intégration pertinents
  • Examiner les suggestions Copilot pour les problèmes potentiels de droits d'auteur (voir Section 9.3)

8.3 Modèles d'automatisation interdits

  • La fusion automatique du code généré par IA sans révision humaine est expressément interdite
  • Utiliser Copilot pour générer des scripts qui automatisent l'accès aux systèmes gouvernementaux sans autorisation appropriée
  • Utiliser Copilot Workspace pour générer et pousser automatiquement des branches de fonctionnalités entières sans révision par le développeur de chaque commit

9Sécurité, chaîne d'approvisionnement et propriété intellectuelle

9.1 Détection des secrets et des identifiants

GitHub Advanced Security (GHAS) doit être activé sur tous les référentiels où Copilot est actif. La Détection des secrets GHAS fournit une détection en temps réel des identifiants commis accidentellement. Les organisations doivent :

  • Activer la Détection des secrets GitHub avec protection des pousses activée (bloque les pousses contenant des secrets détectés)
  • Activer les alertes Dependabot et les mises à jour de sécurité pour toutes les dépendances
  • Appliquer l'analyse du code (CodeQL ou équivalent) comme vérification d'état requise avant la fusion
  • Effectuer des révisions trimestrielles de toutes les alertes GHAS et des délais de résolution

9.2 Désinscription aux données d'entraînement (configuration d'entreprise)

Dans le cadre de l'accord GitHub Copilot Enterprise, les invites de code et les suggestions ne sont pas utilisées par défaut pour entraîner les modèles de GitHub. L'administrateur GitHub Copilot doit confirmer ce paramètre dans la console de politique Copilot de l'organisation :

  • Naviguer vers : GitHub Enterprise → Organisation → Paramètres → Copilot → Politiques
  • Confirmer : « Autoriser GitHub à utiliser mes extraits de code de l'éditeur de code pour améliorer GitHub Copilot » est réglé sur Désactivé
  • Faire une capture d'écran de cette configuration et la conserver comme preuve pour l'ES&A et les fins d'audit

9.3 Propriété intellectuelle et droits d'auteur

GitHub Copilot peut parfois suggérer du code similaire au code open source existant. Les développeurs doivent être conscients des points suivants :

  • Activer le filtre de détection des duplications intégré de GitHub Copilot — il signale les suggestions qui correspondent étroitement au code public et affiche la licence correspondante
  • Ne pas accepter les suggestions signalées comme correspondant aux licences GPL, AGPL ou autres licences copyleft pour utilisation dans des systèmes gouvernementaux propriétaires sans examen juridique
  • Tout code généré par IA incorporé dans des œuvres sous droit d'auteur de la Couronne reste soumis à la Loi sur le droit d'auteur — la Couronne conserve le droit d'auteur; les développeurs doivent documenter les contributions importantes assistées par IA
  • Ne pas soumettre le code gouvernemental aux invites Copilot d'une manière qui divulgue une logique commerciale non publiée ou propriétaire

9.4 Sécurité de la chaîne d'approvisionnement

Copilot peut suggérer des importations de bibliothèques tierces. Avant d'accepter ces suggestions :

  • Vérifier la bibliothèque dans le catalogue de dépendances approuvées de l'organisation
  • Vérifier les CVE connus via Dependabot ou NVD avant la fusion
  • Ne pas ajouter de nouvelles dépendances de packages sans l'approbation du chef d'équipe

10Exigences en matière de langues officielles

🇨🇦 Obligations en vertu de la Loi sur les langues officielles Les institutions fédérales soumises à la Loi sur les langues officielles (L.R.C., 1985, ch. 31) doivent s'assurer que le contenu généré par IA utilisé dans les communications destinées au public ou aux employés est disponible en anglais et en français.

10.1 Commentaires de code et documentation

GitHub Copilot génère des commentaires de code et de la documentation principalement en anglais (la langue par défaut). Pour la documentation, les commentaires en ligne et les chaînes destinées aux utilisateurs dans les systèmes de production, les équipes doivent :

  • Réviser les commentaires générés par l'IA pour leur exactitude avant de s'engager — ne pas se fier à Copilot pour la génération de contenu bilingue sans révision humaine
  • Pour toute chaîne destinée aux utilisateurs ou message système généré par Copilot, s'assurer que les traductions françaises sont produites et révisées par des traducteurs qualifiés ou du personnel bilingue
  • Utiliser les flux de travail de traduction établis; les traductions françaises générées par Copilot ne sont pas considérées comme une traduction officielle aux fins de l'AIPRP ou des communications publiques

10.2 Documentation technique interne

La documentation technique interne (fichiers README, notes d'architecture, wikis de développeurs) générée avec l'aide de Copilot peut être maintenue en anglais pour usage interne des développeurs, conformément à la politique de langue de travail interne de l'institution.

11Surveillance, rapports et audit

11.1 Indicateurs d'utilisation (tableau de bord GitHub Copilot)

L'administrateur GitHub Copilot doit produire un rapport d'utilisation trimestriel au DPI/DSPSI, comprenant :

  • Nombre total de sièges actifs par rapport aux sièges sous licence (taux d'utilisation)
  • Taux d'acceptation des suggestions Copilot par équipe et par référentiel
  • Volumes de requêtes Copilot Chat
  • Tout incident ou événement d'abus signalé
  • Taux d'achèvement de la formation et recertifications en attente

11.2 Conservation des journaux d'audit

Les journaux d'audit GitHub Enterprise capturent toutes les actions administratives (attributions de sièges, modifications de politique, activation des fonctionnalités). Ces journaux doivent être :

  • Conservés pendant un minimum de deux (2) ans conformément aux obligations de gestion de l'information du GC
  • Exportés mensuellement vers le SIEM ou l'archive de journaux sécurisée de l'organisation
  • Disponibles pour les demandes d'accès à l'information, les révisions de sécurité et les audits internes

11.3 Examen annuel de sécurité

La Sécurité des TI doit effectuer un examen annuel du déploiement de GitHub Copilot, comprenant :

  • Réévaluation de la résidence des données et des conditions de service (les accords avec les fournisseurs changent)
  • Révision des alertes GHAS et des vulnérabilités non résolues introduites depuis le dernier examen
  • Révision de toutes les nouvelles fonctionnalités Copilot activées au cours de l'année et leurs implications en matière de sécurité
  • Évaluation mise à jour des menaces et des risques si la posture de risque organisationnelle a changé

12Intervention en cas d'incident

12.1 Incidents à signaler

Les événements suivants doivent être signalés immédiatement à la Sécurité des TI via ACTION REQUISE — insérer le canal de signalement/file d'attente de tickets :

  • Soumission suspecte de données Protégé B ou de RPI à Copilot (injection d'invite accidentelle)
  • Copilot suggérant ce qui semble être des données internes sensibles provenant d'un autre référentiel ou système
  • Suggestions de code malveillant (p. ex., suggestions contenant des portes dérobées évidentes, exfiltration d'identifiants)
  • Utilisation non autorisée de Copilot par une personne non provisionnée
  • Perte ou compromission d'un appareil de développeur avec une session Copilot active

12.2 Procédures d'intervention

  1. Contenir : Révoquer immédiatement la licence Copilot du développeur affecté et l'accès GitHub si une violation est suspectée
  2. Évaluer : La Sécurité des TI examine les journaux d'audit Copilot et l'activité GitHub pour déterminer la portée
  3. Notifier : Si des renseignements personnels ont pu être divulgués, l'Agent de protection des renseignements personnels est notifié dans les 24 heures; notification au CPVP selon les exigences de la Loi sur la protection des renseignements personnels
  4. Remédier : Supprimer tout code commis contenant des données sensibles; faire pivoter les identifiants exposés; émettre des alertes de détection de secrets GitHub
  5. Documenter : Rapport d'incident déposé dans les 72 heures; leçons apprises révisées dans les 30 jours
